# 1.3 PDF-Analyse stabilisieren

**Priorität:** KRITISCH
**Aufwand:** 1 Tag
**Phase:** 1 — Fundament

## Ziel

Ein einziger, zuverlaessiger Weg fuer Agents, PDF-Inhalte zu lesen. Kein Raten, kein Fallback auf "aus dem Gedaechtnis schreiben".

## Hintergrund

Beim Pitch hatte Lava einen "API key error" beim PDF-Lesen und fiel auf einen Fallback zurueck, bei dem sie das Review ohne tatsaechliche PDF-Analyse schrieb. Es gibt aktuell 3 Wege, eine PDF zu lesen:

1. `GET /api/files/{filename}` — Roher Download, Agent muss selbst parsen
2. `GET /api/agents/projects/{projectId}/tasks/{taskId}/attachments/{attachmentId}/content` — Server parst PDF
3. Agent laedt Datei herunter und nutzt lokales `pdf` Tool

Das ist zu viel Auswahl. Agents brauchen genau einen Weg.

## Anforderungen

### Primaerer Weg: Content-Endpoint

```
GET /api/agents/projects/{projectId}/tasks/{taskId}/attachments/{attachmentId}/content
Authorization: Bearer <BYOA Token>
```

Dieser Endpoint existiert bereits. Er muss:

1. **Zuverlaessig funktionieren** — keine API-Key-Fehler, keine Timeouts
2. **Gute Fehlermeldungen liefern** — wenn Parsing fehlschlaegt, klar sagen warum
3. **Im Task Runtime Context integriert sein** — Content wird automatisch im Context mitgeliefert (siehe Task 1.1)
