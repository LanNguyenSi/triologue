# 1.1 Task Runtime Context API

**Priorität:** KRITISCH
**Aufwand:** 2 Tage
**Phase:** 1 — Fundament

## Ziel

Neuer API-Endpoint `GET /api/agents/tasks/{taskId}/context` der einem Agent alles liefert, was er zur Bearbeitung einer Task braucht — in einem einzigen Request.

## Hintergrund

Agents muessen heute 5-10 verschiedene Endpoints aufrufen um eine Task zu bearbeiten (Task lesen, Attachments finden, PDF parsen, Memories laden, Projekt-Infos holen). Das fuehrt zu Fehlern wenn ein Endpoint versagt oder der Agent den falschen waehlt.

## Anforderungen

### Endpoint

```
GET /api/agents/tasks/{taskId}/context
Authorization: Bearer <BYOA Token>
```

### Response-Struktur

```typescript
interface TaskExecutionContext {
  task: {
    id: string;
    title: string;
    description: string;
    priority: string;
    status: string;
    assignedTo: string;
    createdAt: string;
    usedMemoryIds: string[];
  };

  project: {
    id: string;
    name: string;
    description: string;
  };

  attachments: Array<{
    id: string;
    filename: string;
    mimeType: string;
    sizeBytes: number;
    content?: string;       // Server-parsed text content (fuer PDF, Text, Markdown)
    downloadUrl: string;    // Fallback: direkter Download
  }>;

  memories: Array<{
    id: string;
    title: string;
    content: string;        // payload.note
    scope: string;          // PROJECT | GLOBAL
    pinned: boolean;
    tags: string[];
  }>;

  // Actions werden dynamisch aus der Action Registry gebaut.
  // In Phase 1: nur interne Actions (Triologue API).
  // Ab Phase 3: zusaetzlich Connector Actions (SharePoint, Jira, etc.)
  actions: Array<{
    id: string;            // z.B. "task.updateStatus", "sharepoint.upload"
    name: string;          // Menschenlesbarer Name
    description: string;   // Fuer den Agent: was tut diese Action
    type: "internal" | "connector";  // Intern = Triologue API, Connector = externer Service
    method: string;        // "GET" | "POST" | "PATCH" | "PUT" | "DELETE"
    url: string;           // Fertige URL, direkt aufrufbar
    input?: object;        // JSON Schema der erwarteten Parameter
    connectorId?: string;  // Nur bei type=connector: welcher Connector
  }>;

  constraints: {
    maxMessageLength: number;   // 4000
    workflow: string;           // Aus globaler Memory, falls vorhanden
  };
}
```

### Attachment Content Parsing

- **PDF:** Serverseitig mit pdf-poppler oder pdf-parse extrahieren, als `content` String liefern
- **Markdown/Text:** Direkt als `content` liefern
- **Bilder:** `content` bleibt leer, `downloadUrl` verfuegbar
- **Fehler beim Parsen:** `content: null`, `parseError: "reason"` hinzufuegen

### Memories

- Alle PROJECT-scoped Memories des zugehoerigen Projekts laden (archivedAt: null)
- Alle GLOBAL Memories laden
- Sortiert: pinned zuerst, dann nach createdAt desc
- Content aus `payload.note` extrahieren (nicht payload.text)

### Actions

- URLs muessen vollstaendig sein (projectId, taskId, roomId eingesetzt)
- Keine Platzhalter — Agent kann die URL direkt aufrufen
- Auth Header muss der Agent selbst setzen (sein BYOA Token)
- Actions werden aus einer einfachen Action Registry geladen (siehe `actionRegistry.ts`)
- In Phase 1 nur interne Actions registriert:
  - `task.updateStatus` — Task-Status aendern (PATCH)
  - `task.uploadAttachment` — Datei an Task haengen (POST)
  - `room.postMessage` — Nachricht im Room posten (POST)
- Die Registry ist erweiterbar: In Phase 3 registrieren Connectors ihre Actions hier

### Berechtigung

- Nur der zugewiesene Agent (assignedTo == anfragender User) darf den Context abrufen
- Oder: jeder Agent der Participant im zugehoerigen Room ist
- 403 wenn keine Berechtigung

## Dateien die geaendert werden muessen

1. **`server/src/routes/agents.ts`** — Neuen Endpoint hinzufuegen
2. **`server/src/routes/agents.ts`** — Attachment Content Parsing Logik (existiert schon fuer `/content` Endpoint, wiederverwenden)

4. **`server/openapi.yaml`** — Endpoint dokumentieren
