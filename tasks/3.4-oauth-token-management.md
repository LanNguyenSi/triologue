# 3.4 OAuth Token Management

**Priorität:** HOCH
**Aufwand:** 2 Tage
**Phase:** 3 — Enterprise-Integration

## Ziel

Zentrale Verwaltung von OAuth Tokens fuer externe Integrationen (Microsoft Graph, Jira, etc.). Tokens werden sicher gespeichert, automatisch refreshed und nie an Agents weitergegeben.

## Hintergrund

Jede Enterprise-Integration (Teams, SharePoint, Jira) braucht OAuth Tokens. Diese muessen sicher gespeichert werden, automatisch refreshed werden wenn sie ablaufen, und zentral verwaltet werden statt pro Agent.

## Anforderungen

### Prisma Schema

```prisma
model IntegrationToken {
  id            String   @id @default(cuid())
  provider      String                        // "microsoft", "atlassian", etc.
  scope         String                        // "graph", "jira", etc.
  accessToken   String                        // Verschluesselt gespeichert
  refreshToken  String?                       // Verschluesselt gespeichert
  expiresAt     DateTime
  tenantId      String?                       // Azure Tenant ID, Jira Cloud ID, etc.
  metadata      Json     @default("{}")       // Provider-spezifische Infos
  createdBy     String                        // Admin der die Integration eingerichtet hat
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
  
  creator       User     @relation(fields: [createdBy], references: [id])
  
  @@unique([provider, scope, tenantId])
  @@map("integration_tokens")
}
```

### Token Encryption

- Tokens werden vor dem Speichern mit AES-256-GCM verschluesselt
- Encryption Key aus Environment Variable (`INTEGRATION_ENCRYPTION_KEY`)
- Decryption nur im Backend, nie im Frontend oder in API-Responses

### Auto-Refresh

- Background Job prueft alle 5 Minuten ob Tokens bald ablaufen (<10 Min)
- Refresh Token wird genutzt um neuen Access Token zu holen
- Bei Refresh-Fehler: Admin benachrichtigen via Inbox

### Proxy-Pattern

Agents rufen nie direkt externe APIs auf. Stattdessen:
1. Agent ruft Triologue Proxy-Endpoint auf (z.B. `POST /api/integrations/sharepoint/read`)
2. Backend holt Token aus DB, decrypted, setzt Authorization Header
3. Backend proxied Request an externe API
4. Backend gibt Response an Agent zurueck

### Admin UI

- Uebersicht: Welche Integrationen sind verbunden
- Status: Token gueltig / abgelaufen / Refresh fehlgeschlagen
- Actions: Verbinden (OAuth Flow starten), Trennen (Token loeschen)
