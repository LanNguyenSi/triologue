# 3.1 Connector Framework + YAML Format

**Priorität:** HOCH
**Aufwand:** 3 Tage
**Phase:** 3 — Enterprise-Integration

## Ziel

Ein Framework das Connectors deklarativ (YAML) definierbar macht. Proxy Layer der alle Connector-Aufrufe von Agents entgegennimmt, Token injiziert und an den externen Service weiterleitet.

## Hintergrund

90% aller SaaS-Dienste haben REST APIs mit OAuth. Statt fuer jeden einen Custom-Code-Connector zu schreiben, definieren wir Connectors als YAML-Dateien. Das Framework liest die Definitionen, registriert die Actions in der Action Registry und stellt den Proxy Layer bereit.

## Anforderungen

### Connector Definition Format (YAML)

```yaml
# server/src/connectors/definitions/sharepoint.yaml
id: sharepoint
name: Microsoft SharePoint
provider: microsoft
icon: /icons/sharepoint.svg
category: storage              # storage | communication | project-management | crm | custom

auth:
  type: oauth2
  provider: microsoft          # Referenz auf Token Manager Provider
  scope: graph
  requiredScopes:
    - Files.ReadWrite.All
    - Sites.Read.All

actions:
  - id: sharepoint.list
    name: List Files
    description: "Listet Dateien in einem SharePoint-Ordner auf"
    method: GET
    urlTemplate: "https://graph.microsoft.com/v1.0/drives/{{driveId}}/root:{{path}}:/children"
    input:
      type: object
      properties:
        driveId:
          type: string
          required: true
          description: "SharePoint Drive ID"
        path:
          type: string
          default: "/"
          description: "Ordner-Pfad"
    output:
      type: array
      items:
        name: string
        size: number
        lastModified: string

  - id: sharepoint.read
    name: Read File
    description: "Liest den Inhalt einer Datei aus SharePoint"
    method: GET
    urlTemplate: "https://graph.microsoft.com/v1.0/drives/{{driveId}}/root:{{path}}:/content"
    input:
      type: object
      properties:
        driveId: { type: string, required: true }
        path: { type: string, required: true }
    responseType: binary       # Datei-Download, wird serverseitig geparsed

  - id: sharepoint.upload
    name: Upload File
    description: "Laedt eine Datei in SharePoint hoch"
    method: PUT
    urlTemplate: "https://graph.microsoft.com/v1.0/drives/{{driveId}}/root:{{path}}:/content"
    input:
      type: object
      properties:
        driveId: { type: string, required: true }
        path: { type: string, required: true }
        content: { type: string, required: true }
```

### Connector Registry

```typescript
// server/src/connectors/registry.ts

class ConnectorRegistry {
  // Beim Server-Start: alle YAML-Dateien aus definitions/ laden
  loadDefinitions(dir: string): void;
  
  // Connector nach ID holen
  getConnector(id: string): ConnectorDefinition | null;
  
  // Alle verfuegbaren Connectors
  listConnectors(): ConnectorDefinition[];
  
  // Aktive Connectors (mit gueltigem Token)
  listActive(): ConnectorDefinition[];
  
  // Actions fuer ein Projekt zusammenbauen (fuer Task Runtime Context)
  getActionsForProject(projectId: string): ActionDefinition[];
}
```

### Proxy Layer

```typescript
// server/src/connectors/proxy.ts

// Einheitlicher Endpoint fuer alle Connector-Aufrufe
// POST /api/connectors/{connectorId}/actions/{actionId}
// Body: { ...input parameters }

async function executeAction(
  connectorId: string,
  actionId: string,
  input: Record<string, unknown>,
  agentId: string
): Promise<ConnectorResponse> {
  // 1. Connector-Definition laden
  // 2. Berechtigung pruefen (darf dieser Agent diesen Connector nutzen?)
  // 3. Token aus TokenManager holen (decrypted, refreshed)
  // 4. URL Template mit Input-Werten fuellen
  // 5. HTTP Request an externen Service senden (mit Token im Auth Header)
  // 6. Response normalisieren
  // 7. Audit Log schreiben
  // 8. Normalisierte Response zurueckgeben
}
```

### Berechtigungen pro Connector

```prisma
model ConnectorPermission {
  id            String   @id @default(cuid())
  connectorId   String                        // "sharepoint"
  userId        String                        // Agent oder User
  allowedActions String[] @default([])        // Leeres Array = alle Actions erlaubt
  grantedBy     String
  createdAt     DateTime @default(now())
  
  user          User     @relation(fields: [userId], references: [id])
  
  @@unique([connectorId, userId])
  @@map("connector_permissions")
}
```

### Integration mit Action Registry (aus Task 1.1)

Beim Aufbau des Task Runtime Context:
1. Interne Actions laden (task.update, attachment.upload, etc.)
2. ConnectorRegistry fragen: welche Connectors sind aktiv?
3. Fuer jeden aktiven Connector: Actions pruefen ob Agent berechtigt ist
4. Berechtigte Connector-Actions in den Actions-Block einfuegen
5. URLs zeigen auf den Proxy: `/api/connectors/{connectorId}/actions/{actionId}`

### Custom Code Plugins (fuer die 10% Sonderfaelle)

Wenn YAML nicht reicht (GraphQL, Webhooks, Custom Pagination):

```typescript
// server/src/connectors/plugins/jiraPlugin.ts
export class JiraPlugin implements ConnectorPlugin {
  id = "jira";
  
  // Ueberschreibt die Standard-Proxy-Logik fuer bestimmte Actions
  async executeAction(actionId: string, input: any, token: string): Promise<any> {
    if (actionId === "jira.createIssue") {
      // Jira nutzt eigentlich REST, aber fuer komplexe Issue-Typen
      // braucht man Custom-Logik (z.B. Subtasks, Custom Fields)
    }
  }
}
```

## Dateistruktur

```
server/src/connectors/
├── registry.ts              ← Connector Registry
├── proxy.ts                 ← Proxy Layer + API Route
├── types.ts                 ← TypeScript Interfaces
├── definitions/             ← Deklarative Connector-Configs
│   ├── sharepoint.yaml
│   ├── jira.yaml
│   ├── gmail.yaml
│   └── slack.yaml
└── plugins/                 ← Custom-Logik (nur wenn YAML nicht reicht)
    └── jiraPlugin.ts
```
