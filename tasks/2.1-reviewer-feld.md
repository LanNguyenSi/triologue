# 2.1 Reviewer-Feld auf Tasks

**Priorität:** HOCH
**Aufwand:** 2 Tage
**Phase:** 2 — Workflow-Reife

## Ziel

Tasks bekommen ein optionales `reviewedBy` Feld. Wenn eine Task auf `in_review` gesetzt wird, wird der Reviewer benachrichtigt.

## Hintergrund

Im Pitch bat Lan Lava, "@publicplan als Reviewer zu setzen". Lava konnte das nicht, weil Tasks kein Reviewer-Feld haben. Das ist ein Kernbestandteil eines professionellen Workflows.

## Anforderungen

### Prisma Schema

```prisma
model Task {
  // ... bestehende Felder ...
  assignedTo  String?
  reviewedBy  String?          // NEU: User-ID des Reviewers
  
  assignee    User?  @relation("TaskAssignee", fields: [assignedTo], references: [id])
  reviewer    User?  @relation("TaskReviewer", fields: [reviewedBy], references: [id])
}
```

### API Aenderungen

**PATCH /api/projects/{projectId}/tasks/{taskId}:**
- `reviewedBy` als optionales Feld akzeptieren
- Akzeptiert User-ID oder Username (resolve via DB)

**GET Task-Responses:**
- `reviewer` Objekt mitliefern (id, username, displayName) analog zu `assignee`

### Notifications

Wenn `status` auf `in_review` gesetzt wird UND `reviewedBy` gesetzt ist:
1. Inbox-Item fuer den Reviewer erstellen
2. Nachricht im Room posten: "Task 'XYZ' wartet auf Review von @reviewer"

### Frontend

- Task-Detail: Reviewer-Dropdown (analog zu Assignee)
- Task-Card (Kanban): Reviewer-Avatar anzeigen wenn gesetzt
- Task-Edit Modal: Reviewer-Feld hinzufuegen
