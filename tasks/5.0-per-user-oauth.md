# 5.0 Per-User OAuth

**Priorität:** HOCH
**Aufwand:** 3 Tage
**Phase:** 5 — User-Level Integrations
**Abhängigkeit:** 4.0 (OAuth Flow UI)

## Ziel

Jeder User verbindet seine eigenen externen Accounts (SharePoint, Jira). Agents handeln im Kontext des Users der den Task erstellt hat — nicht mit einem globalen Admin-Token.

## Hintergrund

Aktuell (Phase 4): Admin verbindet einmal, ein Token für alle. Problem:
- Agent sieht alles was der Admin sieht
- Kein Berechtigungskontext pro User
- Mitarbeiter A und B haben unterschiedliche SharePoint-Zugänge, aber Agent nutzt immer den gleichen Token

Ziel: Wie Perplexity Computer — jeder User verbindet sein eigenes Microsoft/Atlassian-Konto.

## Anforderungen

### 1. IntegrationToken wird User-spezifisch

```prisma
model IntegrationToken {
  // bestehende Felder...
  userId    String?          // NEU: null = global (legacy), set = user-spezifisch
  user      User?   @relation(...)
  
  @@unique([provider, scope, tenantId, userId])  // Unique pro User
}
```

Migration: Bestehende Tokens bekommen `userId = null` (bleiben global als Fallback).

### 2. User-Settings: "Meine Verbindungen"

Neue Seite: `/settings/connections`

- Gleiche UI wie Admin Connector-Seite, aber:
  - Zeigt nur eigene Verbindungen
  - Kein Admin-Check
  - OAuth Flow speichert `userId` des eingeloggten Users

### 3. Token-Auflösung im Proxy

Proxy-Layer Logik:
1. Task hat `createdBy` → User
2. Suche Token für diesen User + Provider
3. Fallback: globaler Token (userId = null) wenn User keinen eigenen hat
4. Kein Token → 503

```typescript
async function resolveToken(provider: string, scope: string, userId: string): Promise<string | null> {
  // 1. User-spezifischer Token
  let token = await getTokenForUser(provider, scope, userId);
  if (token) return token;
  
  // 2. Fallback: globaler Admin-Token
  token = await getToken(provider, scope);
  return token;
}
```

### 4. Task Runtime Context: User-Kontext

`buildActionsForTask()` erhält den `createdBy`-User des Tasks. Connector-Actions sind nur verfügbar wenn ein Token existiert (user-spezifisch oder global).

### 5. Admin behält Überblick

Admin-Seite (`/admin/connectors`) zeigt:
- Globale Verbindungen (wie bisher)
- Übersicht: welche User haben eigene Verbindungen
- Option: globalen Token als Default setzen

## Akzeptanzkriterien

- [ ] User kann eigene Microsoft/Jira-Verbindung herstellen (ohne Admin)
- [ ] Agent nutzt Token des Task-Erstellers
- [ ] Fallback auf globalen Token wenn User keinen eigenen hat
- [ ] Admin sieht alle Verbindungen (global + per-user)
- [ ] Bestehende globale Tokens funktionieren weiterhin (Migration)
- [ ] User kann eigene Verbindung trennen (revoke)
