# 2.3 Result Router

**Priorität:** MITTEL
**Aufwand:** 2 Tage
**Phase:** 2 — Workflow-Reife

## Ziel

Wenn ein Agent eine Task abschliesst, reagiert die Plattform automatisch: Reviewer benachrichtigen, Summary im Room posten, Inbox-Items erstellen.

## Hintergrund

Heute muss der Agent selbst entscheiden, was nach Abschluss passiert. Das fuehrt zu inkonsistenten Ergebnissen — manchmal postet der Agent eine Zusammenfassung, manchmal nicht. Die Plattform sollte das konsistent uebernehmen.

## Anforderungen

### Trigger

Wenn `status` einer Task geaendert wird auf `in_review` oder `done`:

### Aktionen bei `in_review`

1. **Reviewer benachrichtigen** (wenn reviewedBy gesetzt):
   - Inbox-Item erstellen
   - Optional: Push-Notification (spaeter)
2. **Room-Nachricht posten:**
   ```
   ✅ Task "Screening-Kriterium: Risiken und Go/No-Go" ist bereit fuer Review.
   Bearbeitet von: @lava | Reviewer: @publicplan
   Anhaenge: screening_review_nrw_ki.md (12 KB)
   ```
3. **Audit Log** (wenn Task 1.2 fertig): `task.review_requested`

### Aktionen bei `done`

1. **Ersteller benachrichtigen** (Task Creator oder Projekt-Owner):
   - Inbox-Item erstellen
2. **Room-Nachricht posten:**
   ```
   ✅ Task "Screening-Kriterium: Risiken und Go/No-Go" abgeschlossen.
   Bearbeitet von: @lava | Dauer: 3 Minuten
   ```
3. **Audit Log**: `task.completed`

### Konfigurierbar

Nicht jedes Projekt braucht Notifications. Pro Projekt konfigurierbar:
- `notifyOnReview: boolean` (default: true)
- `notifyOnDone: boolean` (default: true)
- `postSummaryInRoom: boolean` (default: true)

## Abhaengigkeiten

- **Task 2.1** (Reviewer-Feld) fuer reviewer-bezogene Notifications

1. **`server/src/routes/projects.ts`** — Task-Status-Update Hook erweitern
2. **`server/src/services/resultRouterService.ts`** — Neuer Service (erstellen)

4. **`server/prisma/schema.prisma`** — Projekt-Settings fuer Notifications (optional)
