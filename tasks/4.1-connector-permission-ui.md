# 4.1 Connector-Berechtigungen UI

**Priorität:** HOCH
**Aufwand:** 1 Tag
**Phase:** 4 — Connector UI
**Abhängigkeit:** 4.0 (OAuth Flow UI), 3.1 (Connector Framework)

## Ziel

Admin kann pro Agent konfigurieren, welche Connectoren und welche Actions erlaubt sind. Agent sieht in seinem Task Runtime Context nur die Actions für die er Berechtigung hat.

## Hintergrund

`ConnectorPermission` Model existiert (3.1). Der Proxy prüft Berechtigungen. Aber es gibt keine UI um sie zu verwalten — aktuell nur direkt in der DB möglich.

## Anforderungen

### Permission-Verwaltung auf Agent Config Seite

Erweitert die bestehende `/admin/agents/:agentTokenId/config` Seite (2.5):

**Neuer Abschnitt: "Connector-Zugriff"**

- Liste aller verbundenen Connectoren
- Pro Connector: Toggle (erlaubt/nicht erlaubt)
- Pro Connector (wenn erlaubt): Checkboxen für einzelne Actions
  - z.B. SharePoint: ☑ list ☑ read ☐ upload
  - z.B. Jira: ☑ listIssues ☑ getIssue ☑ createIssue ☐ updateIssue

### API Endpoints

```
GET  /api/agents/:agentTokenId/permissions     → Liste aller Permissions
PUT  /api/agents/:agentTokenId/permissions      → Bulk-Update aller Permissions
```

### Task Runtime Context Integration

`actionRegistry.ts` erweitern: Connector-Actions nur zurückgeben wenn Agent die Permission hat.

```typescript
// Vorher (3.1): Connector-Actions werden statisch hinzugefügt
// Nachher (4.1): Nur Actions für die der Agent Permission hat
const connectorActions = await getPermittedConnectorActions(agentUserId);
```

## Dateien

## Akzeptanzkriterien

- [ ] Admin kann pro Agent Connector-Zugriff ein/ausschalten
- [ ] Admin kann pro Connector einzelne Actions erlauben/verbieten
- [ ] Task Runtime Context zeigt nur erlaubte Connector-Actions
- [ ] Agent ohne Permission bekommt 403 beim Connector-Proxy
- [ ] Keine Permission nötig für interne Actions (task.updateStatus etc.)
