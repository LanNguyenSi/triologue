# 4.0 OAuth Flow UI

**Priorität:** HOCH
**Aufwand:** 2 Tage
**Phase:** 4 — Connector UI
**Abhängigkeit:** 3.0 (OAuth Token Management), 3.1 (Connector Framework)

## Ziel

User kann über die UI externe Dienste verbinden (SharePoint, Jira) per OAuth. Ein-Klick-Flow: "Verbinden" → Microsoft/Atlassian Login → Redirect zurück → Token gespeichert.

## Hintergrund

Die OAuth-Token-Verwaltung (3.0) und das Connector-Framework (3.1) existieren im Backend. Es fehlt der komplette Frontend-Flow, damit ein Admin ohne API-Kenntnisse Connectoren einrichten kann.

## Anforderungen

### OAuth Callback Route (Backend)

```
GET /api/admin/integrations/oauth/callback?code=...&state=...
```

- `state` enthält Provider + Scope (z.B. `microsoft:graph`)
- Tauscht Authorization Code gegen Access+Refresh Token
- Speichert verschlüsselt via `tokenManager.storeToken()`
- Redirect zurück zur Connector-UI mit Erfolgsmeldung

### OAuth Start Route (Backend)

```
GET /api/admin/integrations/oauth/start?provider=microsoft&scope=graph
```

- Baut Authorization URL zusammen (tenantId, clientId, redirectUri, scope)
- Redirect zu Microsoft/Atlassian Login
- `state` Parameter für CSRF-Schutz

### Umgebungsvariablen

```env
MICROSOFT_CLIENT_ID=...
MICROSOFT_CLIENT_SECRET=...
MICROSOFT_REDIRECT_URI=https://opentriologue.ai/api/admin/integrations/oauth/callback
ATLASSIAN_CLIENT_ID=...
ATLASSIAN_CLIENT_SECRET=...
ATLASSIAN_REDIRECT_URI=https://opentriologue.ai/api/admin/integrations/oauth/callback
```

### Frontend: Connector-Übersicht

Seite: `/admin/connectors`

- Liste aller verfügbaren Connectoren (aus YAML-Definitionen)
- Pro Connector: Name, Icon, Status (verbunden/nicht verbunden/fehler/abgelaufen)
- "Verbinden" Button → startet OAuth Flow
- "Trennen" Button → revoked Token

### Frontend: Verbindungsstatus

- Grün: Token aktiv, Ablauf > 24h
- Gelb: Token läuft bald ab (< 24h), Auto-Refresh aktiv
- Rot: Token abgelaufen oder Refresh fehlgeschlagen
- Grau: Nicht verbunden
