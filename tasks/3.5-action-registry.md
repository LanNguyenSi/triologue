# 3.5 Action Registry

**Priorität:** MITTEL
**Aufwand:** 2 Tage
**Phase:** 3 — Enterprise-Integration

## Ziel

Plugin-System fuer externe Tools. Jede Integration (SharePoint, Jira, Slack, etc.) registriert ihre Actions in einer zentralen Registry. Der Task Runtime Context baut die verfuegbaren Actions automatisch aus der Registry zusammen.

## Hintergrund

Ohne Registry muss der Task Runtime Context Builder fuer jede Integration hardcoded wissen welche Actions verfuegbar sind. Mit Registry kann jedes Plugin seine Actions deklarativ registrieren, und der Context Builder baut den Actions-Block dynamisch zusammen.

## Anforderungen

### Action Definition

```typescript
interface ActionDefinition {
  id: string;                    // "sharepoint.upload"
  provider: string;              // "microsoft"
  name: string;                  // "Upload to SharePoint"
  description: string;           // Fuer den Agent: was tut diese Action
  method: string;                // "POST"
  urlTemplate: string;           // "/api/integrations/sharepoint/upload?driveId={driveId}&path={path}"
  parameters: ActionParameter[]; // Welche Parameter werden erwartet
  requiresAuth: string;          // "microsoft" → Token aus IntegrationToken
  scope?: string;                // "PROJECT" | "GLOBAL" — wann ist diese Action verfuegbar
}

interface ActionParameter {
  name: string;
  type: string;       // "string" | "file" | "enum"
  required: boolean;
  description: string;
  enumValues?: string[];
}
```

### Registry

```typescript
// server/src/services/actionRegistry.ts

class ActionRegistry {
  private actions: Map<string, ActionDefinition> = new Map();
  
  register(action: ActionDefinition): void;
  unregister(id: string): void;
  getAvailable(projectId: string): ActionDefinition[];  // Filtered by scope + active integrations
  buildContextActions(projectId: string, taskId: string): Record<string, any>;  // Fuer Task Runtime Context
}
```

### Integration mit Task Runtime Context

Der Context Builder (Task 1.1) fragt die Registry: "Welche Actions sind fuer dieses Projekt verfuegbar?" und baut den `actions` Block dynamisch zusammen.

Standard-Actions (updateTask, uploadAttachment, postMessage) sind immer dabei.
Integration-Actions (sharepoint.upload, jira.create) nur wenn die Integration fuer das Projekt aktiv ist.

### Built-in Actions

Diese Actions sind immer registriert:

| ID | Beschreibung |
|----|-------------|
| `task.updateStatus` | Task-Status aendern |
| `task.uploadAttachment` | Datei an Task haengen |
| `room.postMessage` | Nachricht im Room posten |
| `memory.read` | Memories lesen |

## Abhaengigkeiten

- Task 1.1 (Task Runtime Context API)
- Task 3.4 (OAuth Token Management) fuer externe Actions

## Dateien die erstellt werden muessen

1. **`server/src/services/actionRegistry.ts`** — Registry Service (erstellen)
2. **`server/src/integrations/*/actions.ts`** — Jede Integration registriert ihre Actions
3. **`server/src/routes/agents.ts`** — Task Runtime Context nutzt Registry
